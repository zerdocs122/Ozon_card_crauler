# Архитектура масштабируемого сервиса мониторинга позиций

## Задача

500 товаров × 20 поисковых запросов = **10 000 проверок в день**.
Результаты → PostgreSQL. Позиция < 50 → алерт в Telegram. Без блокировок.

---

## Технологии и обоснование

| Компонент | Технология | Почему |
|---|---|---|
| Парсер | Python + nodriver | Реальный Chrome — не детектируется Cloudflare (проверено на практике) |
| Очередь задач | Celery + Redis | Распределение 10 000 задач по воркерам, retry при ошибках |
| База данных | PostgreSQL | Хранение истории позиций, аналитика динамики по времени |
| Алерты | Telegram Bot API | Простая интеграция, мгновенная доставка |
| Прокси | Rotating residential proxy pool | Ротация IP — основная защита от блокировок при масштабировании |
| Планировщик | Celery Beat | Запуск проверок по расписанию (раз в сутки или чаще) |

**Почему не Playwright:** на этапе разработки Playwright блокировался Cloudflare Ozon
даже с `playwright-stealth` и чистым резидентным IPv4. nodriver использует реальный
Chrome пользователя — Cloudflare не видит разницы с живым браузером.

```

└────────┬────────┘
         │
         ▼

    ┌────┴────┐

         ▼

└────────┬────────┘
         │
         ▼

│  Alert Service  │  — проверяет position > 50, шлёт в Telegram
└─────────────────┘
```

## Устойчивость к блокировкам

**Прокси:**
- Пул из 50–100 резидентных (residential) IP — выглядят как обычные пользователи
- Ротация каждые 10–15 запросов на воркер
- Автосмена IP при получении пустой выдачи или кода инцидента Cloudflare

**Rate limiting:**
- Каждый воркер: не более 1 запроса в 3–5 секунд
- Между страницами одного поиска — случайная пауза 1.5–3.5 сек
- Параллельно: не более 5–10 воркеров одновременно
- Итого: ~10 000 задач / 8 часов = ~21 задача в минуту = 2 задачи на воркер

**Браузер:**
- nodriver — реальный Chrome без следов автоматизации
- Уникальный профиль Chrome на каждый воркер
- Случайный viewport, порядок и скорость прокрутки

**Retry-логика:**
- При блокировке: смена прокси + повтор через 5 минут (до 3 попыток)
- После 3 неудач — задача в dead letter queue для ручного разбора

## Схема таблиц PostgreSQL

```sql
CREATE TABLE position_checks (
    id            SERIAL PRIMARY KEY,
    query         TEXT NOT NULL,
    sku           TEXT NOT NULL,
    position      INTEGER,       -- NULL если not_found
    page          INTEGER,
    total_checked INTEGER,
    checked_at    TIMESTAMP DEFAULT NOW()
);

CREATE INDEX ON position_checks (sku, checked_at DESC);
```

## Риски и митигация

| Риск | Вероятность | Митигация |
|---|---|---|
| Ozon обновит антибот | Высокая | nodriver обновляется вслед за Chrome; мониторинг доли пустых выдач |
| Резидентные прокси попадут в блок | Средняя | Несколько провайдеров (Bright Data + Oxylabs), автосмена |
| Капча на выдаче | Средняя | Детектировать по `total_checked = 0`, ставить задачу в паузу на 1–2 часа |
| Ozon изменит структуру URL карточек | Низкая | Мониторинг `total_checked = 0` как сигнал поломки парсера |
| Перегрузка PostgreSQL | Низкая | 10 000 записей в день — копейки, индекс по sku + дата |

## Примерный бюджет в месяц

| Статья | Стоимость |
|---|---|
| VPS для воркеров (2–4 ядра, 4 GB RAM) | ~$20–30 |
| Резидентные прокси (50 GB трафика) | ~$50–100 |
| PostgreSQL (Supabase free tier или managed) | $0–25 |
| Redis (на той же VPS или managed) | $0–10 |
| **Итого** | **~$70–165/мес** |

При минимальном бюджете: своя VPS + датацентровые прокси (~$15/мес) —
выше риск блокировок, подходит для тестового запуска.
